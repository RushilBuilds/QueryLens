# Standing Instructions

## Identity & Voice

- All code comments must be written in first-person engineering voice as if a senior data engineer wrote them (e.g. "I'm using CUSUM here because it handles non-stationary drift better than a simple z-score")
- Never write generic comments like `# initialize the logger` or `# define the function`
- Every function must have a docstring explaining the trade-off made, not just what it does

## Commits

- Never batch unrelated files into one commit
- Each commit message must read like an engineering journal entry (e.g. "add CUSUM detector — chose this over EWMA because it catches gradual drift without tuning a decay parameter")
- Commits should be small and intentional, one logical unit at a time
- Within a milestone, commit and push every 2 sub-parts completed — do not wait until the full milestone is done

## Architecture Decisions

- Every non-obvious architectural choice must be logged in DECISIONS.md with: what was chosen, what the alternative was, and why this was better for this use case

## Code Quality

- No placeholder or TODO code in commits — every commit must be functional
- No AI-sounding variable names like `data_processor` or `helper_function`
- Use precise, domain-specific naming (e.g. `cusum_upper_threshold`, `causal_dag_adjacency`, `fault_injection_rate`)

## What to Never Do

- Never mention Claude, AI, or any tool in any file
- Never write "generated by" anywhere
- Never use generic scaffold language in READMEs
